.NET 9 WPF 多数据库管理工具，**可正常启动运行**，全部代码 **0 错 0 警**。支持 6 种关系库（MySQL/MariaDB/SqlServer/PostgreSQL/Oracle/SQLite）完整功能；MongoDB/Redis/DB2 未实现（UI 已拦截）。

## 三、架构关键点
- **三层**：`DbManager.Common`（加密/日志/常量/错误翻译）、`DbManager.Core`（模型/接口/服务/**抽象层**/仓储）、`DbManager.Wpf`（MVVM 界面）。
- **跨库抽象层（Core/Abstractions + Core/Dialects）——所有跨库差异的唯一收敛点，禁止在 VM 里散写 `switch(dbType)`：**
  - `IIdentifierQuoter`：标识符引用（反引号/方括号/双引号）
  - `IDialect`：分页、限定名、时间/自增/拼接、**DDL（AddColumn/DropColumn/AlterColumn）**
  - `IDbTypeMapper` + `LogicalTypeEnum`：原生↔逻辑类型
  - `DialectProvider`：按 `DbTypeEnum` 取上述抽象
- **元数据**：`IDbMetadataService`（含 schema 维度）+ 5 库实现；`DbTreeNavigateService` 带 `MetadataCache`（懒加载+单节点刷新失效）。
- **本地存储**：SQLite（Dapper），`ConnRepository` 建表 + 字段迁移（迁移失败已容错，不阻断启动）。

## 七点五、本次修复的严重 bug/交互问题
- **双击库节点交互不符预期**：之前关系型库双击会打开一个新的空白 SQL 查询标签页（占满右侧内容区，用户反馈像是"缩小了并跳到陌生页面"），而不是像 Navicat 一样展开看到下面的表；MongoDB 库此前已是展开语义。第一次修复改成手动 `IsExpanded=!IsExpanded`，结果暴露第二个 bug：`IsExpanded` 在 `ItemContainerStyle` 里本就双向绑定，WPF 对 TreeViewItem 头部双击自带"自动切换展开"行为，手动再切一次 = 原生切一次 + 手动切一次抵消，表现为"展开后立刻又缩回去"。最终修复：**去掉手动切换**，交给 TreeViewItem 原生双击行为处理（仍通过双向绑定触发懒加载）。Redis 连接双击打开键浏览器不受影响。
  - **已用 dotnet/wpf 官方源码核实**（非拍脑袋）：`TreeViewItem.OnMouseLeftButtonDown` 里 `if ((e.ClickCount % 2) == 0) SetCurrentValueInternal(IsExpandedProperty, !IsExpanded)` 是双击自动展开的真正出处，`SetCurrentValueInternal` 不会破坏既有绑定，会正确经双向绑定回传到 ViewModel，因此代码里确实不需要（也不能）再手动切一次。
  - **自查顺带发现并修复了一个关联的老 bug（与本次双击改动无关，此前一直存在）**：无论怎么触发展开（点箭头/双击），`TreeViewItem_Expanded`（Expanded 路由事件）和 `OnIsExpandedChanged`（IsExpanded 双向绑定回传）会**同时**各自调用一次 `LoadChildrenAsync`——因为 `IsLoaded` 要等整个异步查询完成才置 true，来不及拦住并发的第二次调用。结果是每次展开节点都会并发发起两次相同的元数据查询（不会有可见的重复数据，因为两次都是整体替换 Children，但白白浪费一次数据库/网络往返）。已加 `_isLoadingChildren` 重入锁修复。
  - **第二轮反馈"双击都不展开了"**：去掉手动切换、完全依赖 TreeViewItem 原生双击展开这条路径，在我本地环境用真实鼠标事件（`SendInput`/`mouse_event` 模拟双击 + UIAutomation 读 `ExpandCollapseState`）反复验证不出稳定复现（怀疑原生双击展开的触发条件与本项目 App.xaml 里全局覆写的 `TreeViewItem` 隐式样式或某些交互细节有出入，未能 100% 查清具体原因）——为避免继续在"猜原生行为到底生不生效"上反复横跳，改为**完全自己接管**：在 `PreviewMouseLeftButtonDown`（隧道阶段）按 `ClickCount==2` 判定双击、手动切换 `IsExpanded`、并 `e.Handled=true` 挡掉后续的原生 Bubble 阶段展开逻辑，不再依赖/寄望于任何"原生自动怎样怎样"的假设；同时用 `FindAncestorTreeViewItem`（沿 `e.OriginalSource` 向上找最近的 TreeViewItem）确保双击嵌套子节点不会被外层节点的同一个 Handler 误接管。
    - ⚠️ 由于自动化模拟鼠标双击在我本地环境总是间歇性失败（窗口被最小化/坐标异常等环境问题，非代码问题），**这次没能做到端到端自动化验证**，务必请人工重新测试；如果双击仍有问题，**点击节点左侧的小箭头（展开图标）来展开是完全独立的另一套原生机制，不受本次改动影响，应该始终可用**，可以先用它验证"树本身能不能正常展开显示表"，再单独反馈双击的具体表现（比如：单击选中是否正常、双击有没有反应、展开一半又缩回等），这样能帮我更快定位。
    - **用户反馈：双击展开已恢复正常。** ✅
- **树启动时是空的，要手动点刷新才显示已保存连接**：`MainWindowViewModel` 之前只在构造时加载连接计数，没有顺带刷新树；已加上启动时自动 `RefreshAsync()`。这次改用 UIAutomation 直接验证（启动应用、不发任何刷新快捷键、轮询查 TreeView 下有没有出现 TreeItem），确认自动加载生效——比双击模拟稳定得多（不涉及鼠标点击时序，只是查询自动化树状态）。
- **SQL 编辑器打开即崩溃**：`Resources/SQL.xshd` 里 `<Begin>#</Begin>` 作为 XML 文本内容声明的正则，AvalonEdit 会用 `RegexOptions.IgnorePatternWhitespace` 编译；该模式下未转义的 `#` 被当成"正则里的行内注释起始符"，导致这条规则的开始正则被吞成空模式、在任意位置零宽匹配，触发 AvalonEdit 死循环保护抛出 `InvalidOperationException`——**与文档内容是否真含 `#` 无关**，任意新建 SQL 查询标签页（哪怕是空文档）必现，双击关系型数据库节点即触发。已改为 `<Begin>\#</Begin>` 转义修复，用最小复现工程（`DocumentHighlighter`+实际 xshd）验证空文档/含 `#` 注释/多种注释混合文本均恢复正常高亮。
